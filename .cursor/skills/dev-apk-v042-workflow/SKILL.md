# Dev APK v0.4.2 Workflow

## 适用场景

- 用户要求按 `v0.4.2` 计划逐项实现。
- 用户要求每个改动自动提交，并持续更新 `Release.md`。
- 用户要求所有项完成后再统一构建 APK。

## 强制执行规则

1. 每完成一个独立子项（A1/A2/.../C9），必须：
   - 修改代码；
   - 更新 `Release.md` 的 `v0.4.2` 对应条目为 `[x]`，并在条目后补充实现 commit 短哈希；
   - 立即执行 `git add -A` 与 `git commit`（提交信息以 `[Cursor]` 开头）。
2. 未全部完成前，不执行最终 APK 构建。
3. 当 `v0.4.2` 所有功能条目均为 `[x]` 后：
   - 执行 `.\gradlew.bat assembleDebug --no-daemon`；
   - 在回复中明确给出 APK 全名与完整路径；
   - 将“构建完成”条目标记为 `[x]`，并提交一次 git。

## 实施顺序

固定顺序执行：

1. A1 -> 2) A2 -> 3) A3 -> 4) B4 -> 5) B5 -> 6) B6 -> 7) C7 -> 8) C8 -> 9) C9 -> 10) 最终构建

## 每项完成后的最小回写格式

在 `Release.md` 的 `v0.4.2` 小节中，将对应行改为：

`- [x] <条目名>。实现 commit：\`<short-hash>\``

## 失败处理

- 若某项无法在当前轮完成，保持 `[ ]`，并在回复中说明阻塞原因与下一步建议。
- 不允许在未实现的情况下提前勾选。
