# 产品需求文档（PRD）— Herdroid

**版本**：0.2（与当前代码对齐）  
**依据**：[Ideas.md](Ideas.md)

**当前实现范围**：OpenAI 兼容对话（`POST …/v1/chat/completions`）；对话根地址由**访问协议、地址、端口**自动拼接；另配置 API Key、模型名；以及与 HA 相关的连接配置。**不包含**语音合成（TTS）或自动朗读。

## 1. 背景与目标

用户在 Android 设备上通过局域网连接 Hermes Agent（HA），在客户端以对话形式与 OpenAI 兼容 API 交互。产品强调界面简洁、配置清晰、连接状态可感知。

## 2. 术语

| 术语 | 含义 |
|------|------|
| HA | Hermes Agent 服务端实例 |
| OpenAI 兼容 API | 提供 `POST /v1/chat/completions` 等端点的服务（可为 HA 或独立网关） |

## 3. 用户画像与场景

- **画像**：需在手机端与 HA/兼容 API 对话、且设备与服务处于同一局域网（或可路由可达）的开发者或高级用户。
- **典型场景**：在设置中填写协议、地址、端口与 API Key、模型名；对话根地址由协议、地址、端口拼接；在主界面发送消息并查看助手回复。

## 4. 用户故事

1. 作为用户，我希望在设置里填写协议、地址和端口，以便指向局域网内的 HA。
2. 作为用户，我希望应用能显示与 HA 的连接状态，以便知道是否在线。
3. 作为用户，我希望在设置中配置协议、地址、端口与 API Key、模型名，以便主界面能按拼接后的根地址发起对话。
4. 作为用户，我希望在主界面进行多轮文本对话并看到历史消息，以便连续交流。
5. 作为用户，我希望界面简洁，以便快速完成配置与日常使用。

## 5. 功能需求

### 5.1 连接 HA（局域网）

- **FR-CONN-1**：应用应支持用户配置的 **scheme（协议）**、**主机名或 IP**、**端口**，并向该端点建立**持续或长连接**（实现方式由架构文档约定，依赖 HA 实际协议）。
- **FR-CONN-2**：应对网络中断、HA 不可达等情况提供可理解的错误状态，并支持用户触发**重新连接**（具体交互见 [UI_UX.md](UI_UX.md)）。
- **FR-CONN-3**：不在本阶段强制要求后台长期保活；若未来需要后台连接，须单独评审并更新 PRD（前台服务、电池策略等）。

### 5.2 设置

- **FR-SET-1**：提供设置界面或等价入口，持久化存储：协议、地址、端口。
- **FR-SET-2**：持久化存储访问 **协议、地址、端口**、**API Key**、**模型名称**；对话使用的根地址由协议、地址、端口推导，不单独持久化第二条 URL（敏感项仅存本地 DataStore，不明文写死）。
- **FR-SET-3**：配置变更后，行为定义：**保存即生效**或**保存后下次连接生效**须在实现前在本文档「待决议」中选定并更新。

### 5.3 主界面与对话

- **FR-CHAT-1**：主界面提供文本输入与消息列表，通过 OpenAI 兼容 API 发送用户消息并展示助手回复。
- **FR-CHAT-2**：对话请求失败时向用户展示可理解的错误信息（网络、HTTP、解析等）。
- **FR-CHAT-3**：支持清空当前会话消息列表（不强制云端同步历史）。

## 6. 非功能需求

| ID | 类别 | 要求 |
|----|------|------|
| NFR-1 | 易用性 | 主路径（配置 → 连接 → 对话）步骤少，文案简短；见 [UI_UX.md](UI_UX.md)。 |
| NFR-2 | 性能 | UI 线程不阻塞；网络与对话在合适协程/线程上执行。 |
| NFR-3 | 安全与隐私 | 见 [PRIVACY_AND_SECURITY.md](PRIVACY_AND_SECURITY.md)。 |
| NFR-4 | 兼容性 | `minSdk`/`targetSdk` 在 [ANDROID_CONVENTIONS.md](ANDROID_CONVENTIONS.md) 中定义并随 Android 版本演进更新。 |

## 7. 验收标准（可测试）

实现阶段用下列条款做冒烟与回归（细则可拆成测试用例）：

1. 用户可保存合法格式的协议、地址、端口及 API Key、模型名，重启应用后配置仍存在，且对话根地址与协议/地址/端口一致。
2. 在 HA 可用且协议匹配时，应用显示**已连接**（或等价明确状态）。
3. 协议、地址、端口与 API Key 有效时，用户可在主界面发送消息并收到助手文本回复。
4. 对话接口不可达或返回错误时，用户能看到明确提示。
5. 设置页「测试连接」「测试对话」在合法配置下能给出成功或失败反馈。

## 8. 开放问题（依赖 HA 与产品决议）

1. **传输协议**：HA 使用 WebSocket、HTTP SSE、gRPC 或其他？消息帧格式与鉴权方式？  
   **影响**：客户端模块划分与依赖库选型。
2. **消息语义**：助手返回的是纯文本、Markdown 还是结构化 JSON？  
   **影响**：展示与后续功能（若增加富文本）。
3. **待决议**：配置保存后是立即重连还是下次启动生效。
