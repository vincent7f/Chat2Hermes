# 产品需求文档（PRD）— Herdroid

**版本**：0.1（文档阶段）  
**依据**：[Ideas.md](Ideas.md)

## 1. 背景与目标

用户在 Android 设备上通过局域网连接 Hermes Agent（HA），实时接收代理返回的消息，并可选择是否自动通过语音合成（TTS）播放。产品强调界面简洁、配置清晰、连接状态可感知。

## 2. 术语

| 术语 | 含义 |
|------|------|
| HA | Hermes Agent 服务端实例 |
| TTS | 文本转语音（Text-to-Speech） |
| 系统 TTS | Android `TextToSpeech` API，使用设备已安装引擎 |
| 网络 TTS | 通过 HTTPS 等调用第三方或自托管 TTS HTTP API |

## 3. 用户画像与场景

- **画像**：需在手机端收听 HA 回复、且设备与 HA 处于同一局域网的开发者或高级用户。
- **典型场景**：配置 HA 地址后进入主界面；开启自动播报后，HA 推送的文本由 TTS 朗读；在安静环境可一键关闭播报。

## 4. 用户故事

1. 作为用户，我希望在设置里填写协议、地址和端口，以便指向局域网内的 HA。
2. 作为用户，我希望应用能显示与 HA 的连接状态，以便知道是否在线。
3. 作为用户，我希望在主界面一键开关「自动 TTS 播报」，以便控制是否朗读 HA 返回内容。
4. 作为用户，我希望选择系统 TTS 或网络 TTS，以便在音质、延迟与隐私之间取舍。
5. 作为用户，我希望界面简洁，以便快速完成配置与日常使用。

## 5. 功能需求

### 5.1 连接 HA（局域网）

- **FR-CONN-1**：应用应支持用户配置的 **scheme（协议）**、**主机名或 IP**、**端口**，并向该端点建立**持续或长连接**（实现方式由架构文档约定，依赖 HA 实际协议）。
- **FR-CONN-2**：应对网络中断、HA 不可达等情况提供可理解的错误状态，并支持用户触发**重新连接**（具体交互见 [UI_UX.md](UI_UX.md)）。
- **FR-CONN-3**：不在本阶段强制要求后台长期保活；若未来需要后台连接，须单独评审并更新 PRD（前台服务、电池策略等）。

### 5.2 设置

- **FR-SET-1**：提供设置界面或等价入口，持久化存储：协议、地址、端口。
- **FR-SET-2**：提供 TTS 相关选项：**引擎类型**（系统 / 网络）、网络 TTS 的 **Base URL** 与**必要鉴权占位**（若采用 API Key，仅存于本地 DataStore，不明文写死）。
- **FR-SET-3**：配置变更后，行为定义：**保存即生效**或**保存后下次连接生效**须在实现前在本文档「待决议」中选定并更新。

### 5.3 主界面与 TTS

- **FR-TTS-1**：主界面提供**一键开关**：开启时对 HA **返回的文本消息**执行 TTS；关闭时仅展示文本（不播放）。
- **FR-TTS-2**：系统 TTS：使用 Android 官方 `TextToSpeech`；需处理引擎未就绪、语言包缺失等情况，并向用户提示。
- **FR-TTS-3**：网络 TTS：通过可配置的 HTTP(S) 端点将文本送出并获取音频或流；失败时**降级**为系统 TTS（可配置「不降级仅报错」为可选项，待决议）。
- **FR-TTS-4**：对连续多条消息应定义**队列或打断策略**（例如新消息打断当前朗读，或排队），默认策略在实现前写入架构文档并在此引用。

## 6. 非功能需求

| ID | 类别 | 要求 |
|----|------|------|
| NFR-1 | 易用性 | 主路径（配置 → 连接 → 开关播报）步骤少，文案简短；见 [UI_UX.md](UI_UX.md)。 |
| NFR-2 | 性能 | UI 线程不阻塞；消息与 TTS 调度在合适协程/线程上执行。 |
| NFR-3 | 安全与隐私 | 见 [PRIVACY_AND_SECURITY.md](PRIVACY_AND_SECURITY.md)。 |
| NFR-4 | 兼容性 | `minSdk`/`targetSdk` 在 [ANDROID_CONVENTIONS.md](ANDROID_CONVENTIONS.md) 中定义并随 Android 版本演进更新。 |

## 7. 验收标准（可测试）

实现阶段用下列条款做冒烟与回归（细则可拆成测试用例）：

1. 用户可保存合法格式的协议、地址、端口，重启应用后配置仍存在。
2. 在 HA 可用且协议匹配时，应用显示**已连接**（或等价明确状态）。
3. 开启自动 TTS 时，模拟或真实 HA 消息到达后，用户可听到与文案一致的朗读（系统 TTS 路径至少打通）。
4. 关闭自动 TTS 时，消息仍可见，但不触发播放。
5. 切换为网络 TTS 并配置有效端点时，能完成一次成功播报；端点不可达时有明确错误提示，且行为符合架构中定义的降级策略。

## 8. 开放问题（依赖 HA 与产品决议）

1. **传输协议**：HA 使用 WebSocket、HTTP SSE、gRPC 或其他？消息帧格式与鉴权方式？  
   **影响**：客户端模块划分与依赖库选型。
2. **消息语义**：HA 返回的是纯文本、Markdown 还是结构化 JSON？是否需要 strip 标签再 TTS？  
   **影响**：TTS 前处理管道。
3. **网络 TTS 契约**：具体 REST 路径、请求体、音频格式（PCM/WAV/OGG）、流式与否？  
   **影响**：网络 TTS 适配器实现。
4. **待决议**：配置保存后是立即重连还是下次启动生效；网络 TTS 失败时是否强制降级。
